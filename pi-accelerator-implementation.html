Snowflake & Data

The core data platform layer — querying, exploration, marketplace, semantic models, Cortex Analyst, governance, lineage, data quality, and cost analysis. This domain extends the existing SQL workspace into a complete Snowflake operating environment.

Write, Validate & Execute SQL Queries
Extend

The foundation already exists. Enhancement adds multi-statement execution, transaction control, query plan visualization, and real-time query profile analysis directly in the panel.

New MCP Tools
explain_query — Calls EXPLAIN on any SQL, returns the query plan as structured JSON for visualization.
get_query_profile — Fetches the QUERY_HISTORY + QUERY_ATTRIBUTION for a given query_id including operator-level stats.
validate_sql — Dry-run validation without execution using EXPLAIN USING TABULAR, returns errors and warnings.
run_transaction — Executes multi-statement blocks with BEGIN/COMMIT/ROLLBACK management and partial rollback on error.
format_sql — Server-side formatting using sql-formatter with dialect-specific rules applied.
UI Components
Query plan tree visualizer — renders EXPLAIN output as a collapsible node tree with cost highlighting on expensive operators.
Inline error overlay on the SQL editor — squiggles under offending tokens from validate_sql response.
Query profile mini-panel — shows bytes scanned, partitions pruned, spill, and operator breakdown without leaving the workspace.
Multi-tab SQL editor — manage multiple queries simultaneously, each with independent execution context.

Task Planning & Multi-Step Workflows
New

The platform's highest-leverage capability: describe a complex data engineering goal in one sentence → AI generates a complete multi-step plan, executes each step using the appropriate domain skills, and tracks progress through a task board.

Implementation Architecture
Planner Agent — a meta-skill that receives a high-level goal, decomposes it into ordered sub-tasks, assigns each sub-task to the correct skill domain, and tracks completion.
Task Queue — each decomposed step is a queued task with: assigned skill, status (pending/active/done/blocked), input context, and output artifact link.
Cross-skill chaining — output of one step becomes input context for the next. E.g., schema design step outputs DDL → development step uses that DDL as context for model generation.
Human-in-the-loop gates — configurable approval checkpoints between steps. Blocks on "Confirm this DDL before execution?" type prompts.
Example Workflow
Goal: "Set up a pipeline that loads customer orders from S3 into Snowflake, models them in dbt, and creates a Streamlit dashboard."
Step 1 (Ingestion): Generate stage + COPY INTO SQL for S3 → Snowflake. Execute with approval. ✓
Step 2 (Architecture): DESCRIBE loaded table → generate dbt staging model + schema YAML + tests. ✓
Step 3 (Testing): Run generated dbt tests. Report quality scorecard. ✓
Step 4 (App): Generate Streamlit app using the mart model. Deploy to Snowflake. ✓
UI Components
Workflow Board — Kanban-style view of task queue: Planned / Running / Awaiting Approval / Done. Each card links to the artifact it produced.
Plan preview before execution — AI shows the decomposed steps for user approval before any MCP tool is called.

Backend extension pattern

Every new MCP tool follows the existing ToolRegistry._register(ToolDefinition, handler) pattern. New domains are organized as separate registry modules (e.g., cortex_registry.py, dbt_registry.py, security_registry.py) that are instantiated and merged into the main registry at startup.
